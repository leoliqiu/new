Read input and background sources

The skill starts by extracting only the facts supplied by the user, including relevant facts from Jira stories, high-level background, Markdown process docs, or a provided background directory. It does not invent login data, statuses, expected messages, test records, URLs, or application behavior.

  • Capture Jira fields, actors, actions, pages, forms, records, source files, and outcomes.
  • Use Markdown headings, lists, and tables to identify process flow, access options, parameters, and validations.
  • Summarize which background files or source cases informed the output.
  • Separate known facts from assumptions.
  • Mark missing required details as Needs clarification.

qTest Steps Provide step and validation columns, ready for qTest-style entry.

What Makes a Good Step?

Weak Step Better qTest Step
Click Submit. As the Requester, complete the form review page and submit the request using the provided values.
Verify success. The request is submitted, a confirmation message appears, and the request status changes to the expected submitted status.
Approve request. As the Approver, open the submitted request, review all required sections, enter the approval decision, and save the decision.

Explore the Skill Rules

The skill uses only what the user provided. Missing information is not guessed. It is either marked as Needs clarification in the draft or asked about after the test case.

When the same test can be run with different users, setup steps, access-management choices, workflow paths, branches, validations, or repeated field values, the skill uses an option table to show those run choices clearly.

When qTest parameters and values are provided, the skill checks the draft against them: every used parameter needs provided values, and every provided parameter is either used or called out as provided but not used.

When the input contains multiple source test cases, the skill first tries to consolidate related cases into one qTest case using options and parameters. If the flows are independent or incompatible, it splits them into separate qTest cases.

When the user provides Jira stories, high-level background, Markdown process docs, or a directory of background information, the skill uses them as supporting context, records the sources used, and asks for clarification when sources conflict or key details are missing.

The validation column must prove a real result: a status changed, a record was created, a message appeared, data persisted, a workflow moved forward, or an error was displayed.

Clarification questions are grouped by type: qTest structure, step process, users/login, test data, validation, or other process. This helps the user answer the exact gaps blocking a better test case.

Review Export Process

Use this process to audit current qTest exports and produce findings, recommended corrections, revised test cases, and focused clarification questions.

1

Sources

Read the export and all supplied supporting information.

2

Inventory

Identify every exported test case and business flow.

3

Completeness

Check description, scope, preconditions, users, and data.

4

Execution

Confirm each step gives a reasonable amount of instruction.

5

Results

Check validations, parameters, options, and access variants.

6

Review Output

Present findings first, then revised qTest content.

Read the export and supporting context

Start with the exported qTest content, then use Jira stories, Markdown process docs, requirements, parameter lists, and other provided sources to understand what the case is intended to test.

  • Preserve the meaning of the exported test case.
  • Map supporting sources to the cases they affect.
  • Do not invent missing values or behavior.

Review Quality Gates

Area Review Question
Description Does it explain the purpose, process, scope, starting point, and intended outcome?
Preconditions Are all necessary accounts, access, records, data, environment, configuration, and prior workflow states documented?
Test Steps Can another tester follow each step without guessing, while avoiding unnecessary click-by-click detail?
Validation Does every step verify a concrete and observable result?
Users and Access Are all roles, login needs, setup differences, and access-management options clear?
Parameters Does every referenced parameter have values, and is every provided parameter used or called out?

Finding Severity

Severity Meaning
Critical The test cannot be executed or its result cannot be determined.
High Important setup, action, access, parameter, or validation information is missing.
Medium The case is executable but contains ambiguity, weak organization, or insufficient detail.
Low Clarity, consistency, naming, or formatting improvement.

Required Review Deliverables

Deliverable What It Contains
Review Summary Case count, overall readiness status, and finding counts by severity.
Test Case Inventory Each exported case, its business flow, review status, and consolidate/keep-separate recommendation.
Findings Severity, affected case, problem, impact, and recommended correction.
Step-Level Review Step-specific assessment with recommended action text and expected result.
Revised qTest Cases Corrected descriptions, necessary preconditions, users/access, parameters, steps, and validations supported by the sources.
Clarifications Only the unanswered questions required to finish the review without inventing data.
Execution-detail standard: a step should contain enough instruction for a tester unfamiliar with the process to complete one concrete task without guessing, but it should not fragment a single task into unnecessary click-by-click steps.
